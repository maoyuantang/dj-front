# API 与联调规则

OpenAPI 是接口结构的优先契约来源。写类型和请求前先定位真实 operation；没有可验证契约时，由接口负责人提供 method、path、参数、响应、错误和鉴权语义，不能从 UI 或行业惯例猜字段。

业务 API 必须复用 `src/api/request.ts` 和集中 env，在 API 边界完成传输壳到领域类型的映射。页面不直接调用底层 `uni.request`，也不各自解析 `response.code`。loading、成功、已知业务错误、401/403、超时/网络、取消和重复提交要有明确策略；提交类请求默认不自动重试，取消/去重 key 必须稳定并清理。

当前 Bearer、401 清理和登录跳转是初始化默认值，首次真实联调必须按后端行为复核。Token、Authorization、个人数据和完整敏感响应不得进入日志、截图或文档。Mock 只隔离测试边界并明确标识，不能冒充生产契约。

联调检查包括 operation 来源、字段映射、Network 脱敏证据、错误恢复、loading 复位和重复提交。正例：API 层把已验证响应映射为领域类型。反例：根据页面字段名编造 endpoint 和成功码。
