# 全局规则

## 适用范围

代码、文档、配置、诊断和 Review 始终适用。专项规则可以增加约束，但不能静默取消安全、证据和用户改动保护边界。

## 开始前

1. 复述目标、范围、非目标、假设和可验证的成功标准。
2. 有多个合理解释时集中询问；附仓库证据、推荐方案、替代和延期影响，不猜。
3. 修改接口前读取目标文件、直接调用方、导出、公共工具、配置和最近测试。
4. 检查工作区，区分用户已有改动和当前任务改动；允许只读 Git 检查。

## 实现纪律

- 选择满足目标的最小正确实现，不添加预测性功能，不为单次使用提前抽象。
- 只改必须触及的文件，不顺手整理邻近代码、注释和格式；保护用户改动。
- 遵循仓库现有命名、结构和风格。模式冲突时选择证据更强、更新或验证更充分的一种并说明，不拼成含糊的第三种。
- 高风险阶段前 checkpoint，能够说明已完成、已检查、剩余和恢复方式；失去上下文时先停下重述。
- 依赖只在已批准计划或用户明确要求时增加；日常功能任务不擅自升级、删除或换库。

## 验证与汇报

默认可运行与改动相关的 type-check、只检查式 lint/stylelint/format、unit test 和 build；会自动修复文件的命令必须明确说明。不得自动启动常驻 dev server、操作微信开发者工具、上传或发布。验证与风险相称，跳过、失败和平台人工项必须分别列出。

禁止泄露凭证、虚构事实/命令/UI 验收、执行未经要求的 Git 写操作、覆盖用户改动和任务外重构。正例：追踪调用链后只修改目标 API 与类型并运行相关检查。反例：顺手格式化全仓库，或 H5 build 成功后宣称微信真机已通过。
